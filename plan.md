# forgenotes

## Идея проекта

Это проект который представляет себя как CLI / TUI / WebUI приложение для заметок
учебы с возможностью работать с ИИ как помощником в них.

- Для чего это нужно? 

Расмотрим вриант если мы имеем просто заметки мы можем не понимать их
настоящие качество мы можем не слишком хорошо их сортировать но с помощю ```LLM``` можно вызвать 
нужный tools и он отсортирует их сделает текст в них болле структурированым, понятным и правильным

## Архитектура приложение

Теперь начем разбирать как будет выглядить архитектура проекта
начиная с самого нижнего уровня до болле абстрактных слоев программы.

### Как будут храниться файлы (замтеки)

#### Основная философия хрнение заметок
	- Все храниться в json файлов.
	- Каждая сущность (заметка, тест, словарная запись, attachment) имеет свой UUID как имя файла.
	- в json должны храниться метаданные теги, ссылки, вложение.
	
Следуя этой философий хранение файлов
таким образом у нас будет не просто голый ```markdown```
а еще много дополнительных методанных.
Все данные которые у тебя есть свазанны между собой.

##### Плюсы хранить метаданных
	- Теги
	    - Быстрый пойск заметок с похожими данными.
	    - Возможность сортеровать програмно заметки.
	- Сылки
	    - Дает плюсы такие же как у ```Гипер текста```.
	- Вложение
	    - можно будет добовлять медиа данные которые тоже будут
		помогать понимать материал заметок.
	- Тесты
	    - Сразу найти тест для этой заметки которая будет подходить
		под тему заметки.

- Вот струтура json заметки

```json
{
    "content": "сдесь храниться голый markdown текст заметок.",
    "title": "титл заметки (имя заметки).",
    "uuid": "uuid замтеки ведь титл может быть одинаковым.",
    "description": "Описание заметки",

        "tags": ["название предмета", "тема предмета"], // нужно для быстрого пойска заметки для ```LLM``` чтоб не траить токены на просмотр content заметок
        "links": ["uuid дргуйх заметок."], // чтоб можно было ссылаться на другие заметки

```

## Тесты

У нас будет сборник вопросов для определенной замтеки
чтоб проверить запомнил / знаешь ли ты материал из заметки.

Это тоже будет json с разными типами вопросов.

Типы вопросов.

1. вопрос где есть текст самого вопроса и список возможных ответов
    и список ответов что позволит сделать так что можно выбрать одновременно
    несколько ответов хорошо например для иероглифов которых можно по разному
    перевести (есть несколько перевод и надо проверить знаешь ли ты их все?).
    
    - плюсы
	1. можно проверать правилно ли ответил школьник / студент без ИИ тоесть програмно.
        2. очень ефективно для запоминание терминов, произношений и переводов.
    - минусы
	2. далеко не разгуляещя если над задачой надо думать то не подходит.


2. вопрос где тебе надо написать ответ в вручную .

    - плюсы
	1. можно отвечать на вопросы которые требует не просто на дроченых знаний
	а понимание темы.

    - минусы
	1. нельза проверить ответ без ИИ.


3. вопрос но тебе надо написать ответ вручную и он должен совподать польносты
   тоесть у наснету список возможных ответов но есть точный ответ.

   - плюсы
	1. для математики где надо написать точное значение  очень подходит.
	2. нельзя догодаься или же методо исключение вывести правельный ответ
	3. можно проверать сочинение текста перевод предложений.
   - минусы
	3. одна опечатка и все ты не правильно ответил
	но для этого будет кнопка что ты правильно ответил просто опчатка
	или попросить ИИ подвердить что твой ответ валидный и это просто
	опечатка.

##  Для чего можн использовать ```LLM```

### Философия LLM в этом приложение
    - Все можно сделать в ручную но и можно в чате попросить LLM сделать
	какоелибо дестивя сгенерироать заметку, тест, перевод.
    - LLМ работает со всеми другими инструментами через MCP.
 
### Интеграция LLM в приложение (через MCP)

**ForgeNotes** использует **Model Context Protocol (MCP)**, чтобы LLM могла безопасно и эффективно работать с вашими заметками, тестами и словарными записями.

#### Как это работает
В приложении поддерживаются **чаты (сессии)** с сохранением контекста. В рамках одной сессии вы можете просить модель:

- Найти и показать нужный материал из заметок
- Сделать заметки более понятными, структурированными или подробными
- Сгенерировать тесты или словарные записи
- Отсортировать/переработать материал
- Добавить новые знания в заметки

**Важное правило**: модель **всегда** сохраняет важные результаты (новые заметки, тесты, переводы, улучшенные структуры) в формате Markdown или напрямую через MCP-инструменты.  
Со временем модель запоминает ваши предпочтения: стиль заметок, приоритетные темы и как вы любите оформлять материал.

#### Доступные MCP-сервера

1. **note-mcp** — основной сервер для работы с заметками  
   Позволяет модели читать, создавать, обновлять и управлять заметками.

   **Основные инструменты:**
   - `read_note(uuid)` — возвращает только чистый Markdown-текст заметки (экономит токены)
   - `read_note_json(uuid)` — возвращает полный JSON заметки (включая все метаданные)
   - `read_title(uuid)`, `read_tags(uuid)`, `read_links(uuid)` и другие «read_*» инструменты — для быстрого получения отдельных частей без загрузки всего файла
   - Полноценный **CRUD**:
     - Создание и обновление заметок
     - Добавление/удаление тегов, ссылок (links), тестов, словарных записей и вложений

2. **tests-mcp** — сервер для работы с тестами  
   Даёт доступ к созданию, чтению, обновлению и удалению тестов, привязанных к заметкам.  
   Модель может генерировать новые вопросы разных типов (multiple choice, open answer, exact answer) и сохранять их.

3. **dictionary-mcp** — сервер для словарных записей  
   Позволяет управлять переводами, значениями слов/иероглифов, примерами использования и связанными метаданными.

#### Преимущества такого подхода
- **Экономия токенов** — модель запрашивает только нужные данные (например, только теги или только content).
- **Безопасность и контроль** — LLM не имеет прямого доступа к файловой системе, всё идёт через явно определённые инструменты.
- **Гибкость** — модель может самостоятельно решать, когда и какой инструмент вызвать (например, сначала поискать по тегам, потом прочитать конкретную заметку).
- **Персистентность** — всё, что модель создаёт или изменяет, сохраняется в вашем vault и остаётся доступным даже после перезапуска приложения.

В будущем можно будет легко добавлять новые MCP-сервера (например, для attachments, spaced repetition или экспорта).

# Общая архитектура слоев програмы.

В начале идут input / output интерфэйсы для пользователя.
такие как CLI, TUI, WebUI в которых нету логики приложние (Кроме отоброжение интерфэйса и ввода данных).
они делают запросы в нужные сервисы которые сами аркестрируют другими модулями / сервисами.
такой подход не дает пользовательскому слою работать с ядром / голыми файлами на прямую толька на уже прописаных сценариев.
А сервисы уже оброщаються в ядру в котором есть толька логика CRUD и проверка валидность входных данных.

CLI / TUI / WebUI передает команда с данными который ввел пользователь.
будет отдельный сервис для каждой команды

Сервис например для создание теста.
В начале проверает валидность всех входных данных.
проверает что нужно сделать если все данные уже предоставленны 
то просто обрщаеться к ядру чтоб добавить в репозиторий файл теста
добавить uuid теста в заметку тоже через ядро.

Если например в параметрах указан флаг создать тест с помощью **LLM** то сервис
возьмет текст заметки
и проверить есть ли промпт в входных параметров на что именно из заметки упоминать
в тесте
если нету то проверить есть ли уже у заметки тесты возьмет их описание или весь json
и соберет все данные в один промпт чтоб ИИ не делала похожие тесты они были
по желание пользователя

Вот схема.

```
CLI / TUI / WebUI
      ↓  ↑
    Services  ←→ Others services
       ↓             ↓
      Core    ←      ←

```

## LLM provider

Общий обьект для ответа LLM.

```rust
/// Represents a standard response from an LLM, enriched with
/// additional details such as metadata and contextual information.
pub struct LlmGenerateResponse {
    /// Plain llm response text.
    pub content: String,
    /// used model name
    pub model: String,
    /// Tokens usega info
    pub token_usage: TokenUsage,
}

/// Contain information about usage of tokens.
pub struct TokenUsage {
    /// Input tokens count.
    pub prompt_tokens: u32,
    /// Output tokens count.
    pub completion_tokens: u32,
    /// Total tokens usegae.
    pub total_tokens: u32,
}
```

Общий обьект для запроса в LLM проваидера.

```rust
/// Standart reuqest object to llm provider generate method
pub struct LlmGenerateRequest {
    /// Model system prompt 
    pub system_prompt: String,
    /// Model prompt
    pub prompt: String,
    /// Model temperature parameters
    pub temperature: f32,
}
```

Общий интерфэис для все LLM провайдеров.

```rust
#[async_trait]
pub trait LlmProvider {
    /// Standart LLM provider interface

    fn name(&self) -> &'static str;
    
    async fn generate(&self, request: LlmGenerateRequest) -> Result<LlmGenerateResponse, LlmError>;
}
```
